# refund-agno — put an LLM agent on real refunds, safely

This example is the headline pitch demo for KIFF. It puts an Agno agent on a
real, consequential job — issuing customer refunds — and shows what makes that
shippable. It runs the same agent twice over three support tickets:

1. **Run A — without KIFF.** The agent's tool mutates a mock database
   directly. Small refunds and a $999 one all go through, because nothing
   sits between the model and the money. This is what most agentic backends
   ship today.
2. **Run B — through KIFF.** Same agent, same model, same prompt, same
   tickets. Now the tool POSTs to a tiny KIFF HTTP server, and the agent can
   actually do the work: the eligible refund executes straight through
   (`AUTO_REFUND`); the high-value refund waits for a human (`approval_required`);
   an operator grants one and denies another; the granted retry executes; the
   denied one stays put.

The story ends with the audit timeline and a rebuild check that proves the
materialized state matches the events — the proof you can verify after the fact.

The offline run is hermetic. No AWS credentials, no Bedrock, no API keys.
This is intentional: a CTO can clone the repo, run `make demo`, and watch an
agent ship refunds inside the lines in under sixty seconds.

The Bedrock run uses Agno's native AwsBedrock provider with whichever
model id you set in `BEDROCK_MODEL_ID` (the user-side run that produced
the transcript below used `moonshotai.kimi-k2.5`). It is the same path
`the-line/apps/agent-runtime` runs on.

## Prerequisites

- Go 1.22 or newer
- Python 3.10 or newer
- For `make demo-bedrock`: AWS credentials with Bedrock access and
  `AGNO_MODEL_PROVIDER=bedrock`. Provide them in
  `kiff-framework/.env` (preferred — shared across examples) or in
  `examples/refund-agno/agent/.env`. The Makefile loads the framework
  `.env` first, then the per-example `.env`. **Use placeholders only in
  any committed file; never commit real credentials.**

The Python side imports `agno` and `boto3` lazily (only when the Bedrock
provider is requested), so `make demo-offline` and `make test` work
without any Python dependencies installed.

## Quickstart

```bash
cd examples/refund-agno
make demo            # alias for make demo-offline (deterministic, no AWS)
```

That will:

1. compile `bin/refund-agno-server`
2. start it on a free port
3. run the unguarded agent against a mock DB
4. run the same agent through KIFF (auto-grant one approval, auto-deny
   another)
5. print the audit timeline and rebuild check for every order
6. shut the server down

To run with **real Agno + AwsBedrock inference** (the same path
`the-line/apps/agent-runtime` runs on):

```bash
# Provide credentials in kiff-framework/.env (preferred — shared across
# examples) or examples/refund-agno/agent/.env. Required keys:
#
#   AGNO_MODEL_PROVIDER=bedrock
#   AWS_REGION=<your-region>
#   BEDROCK_MODEL_ID=<a-bedrock-model-id>
#   AWS_ACCESS_KEY_ID=<...>
#   AWS_SECRET_ACCESS_KEY=<...>
#
# Then:
make demo-bedrock
```

The output header for each run will read:

```
[run-no-kiff]   provider=bedrock model=<your-model-id> real_inference=true
[run-with-kiff] provider=bedrock model=<your-model-id> real_inference=true
```

and every per-ticket line carries a `proposal source` tag —
`offline_fixture` or `bedrock:<model_id>` — so it is never ambiguous
whether you are looking at a real LLM proposal or the deterministic
stub.

To run only tests:

```bash
make test
```

## Offline vs. Bedrock

| Aspect              | offline                                  | bedrock                                                 |
| ------------------- | ---------------------------------------- | ------------------------------------------------------- |
| Provider            | Deterministic Python fixture             | Agno `Agent(model=AwsBedrock(...))` with structured output |
| Reproducible        | Yes (identical bytes every run)          | No — model output varies                                |
| Network calls       | None                                     | Yes — AWS Bedrock                                       |
| AWS creds required  | No                                       | Yes                                                     |
| Reasoning quality   | Two-line stub                            | Multi-sentence model reasoning                          |
| Confidence          | Always 0.78                              | Self-reported by the model (typically 0.5 – 0.9)        |
| Tool argument shape | Hard-coded from the ticket               | Generated by the model from the prompt                  |
| What changes in KIFF | Nothing                                 | Nothing                                                 |

The KIFF behavior is identical in both modes: small refunds auto-execute,
large refunds open approvals, granted approvals execute, denied
approvals stay blocked, the timeline and rebuild always reconcile. The
provider only changes _who proposes_ — the runtime layer that validates
and gates the proposal is unchanged.

## Live Bedrock demo (illustrative)

Run it with:

```bash
make demo-bedrock
```

The transcript below is **illustrative, not exact**. Reasoning text and
self-reported confidence vary between runs because the proposal is
produced by a real LLM. KIFF's runtime behavior is stable: validation,
routing, and approval rules are deterministic and depend only on the
amount, the state, and the approval store — never on the model's words.

```text

========================================================================
[run-no-kiff]   provider=bedrock model=moonshotai.kimi-k2.5 real_inference=true
[run-with-kiff] provider=bedrock model=moonshotai.kimi-k2.5 real_inference=true

[ticket ticket-1] order=order-1 amount=4200 cents
    proposal source : bedrock:moonshotai.kimi-k2.5
    agent reasoning : The customer received the wrong size and provided photo proof…
    agent confidence: 0.85
    KIFF outcome    : executed           action=AUTO_REFUND
    KIFF state      : REFUNDED

[ticket ticket-2] order=order-2 amount=99900 cents
    proposal source : bedrock:moonshotai.kimi-k2.5
    agent confidence: 0.60
    KIFF outcome    : approval_required  action=REFUND_ORDER
    KIFF approval id: approval-order-2-1
    KIFF state      : PAID

[run-with-kiff] granted approval-order-2-1 → retrying ticket ticket-2
    KIFF outcome    : executed           action=REFUND_ORDER
    KIFF state      : REFUNDED

[run-with-kiff] denied approval-order-3-2 → retrying ticket ticket-3
    KIFF outcome    : approval_required  action=REFUND_ORDER
    KIFF state      : PAID

  rebuild(order-1): materialized='REFUNDED' replayed='REFUNDED' events=3 ✓
  rebuild(order-2): materialized='REFUNDED' replayed='REFUNDED' events=3 ✓
  rebuild(order-3): materialized='PAID'     replayed='PAID'     events=2 ✓
```

Proof points to look for in your own Bedrock run:

- Header lines say `provider=bedrock` and `real_inference=true`.
- Each ticket prints `proposal source : bedrock:<your-model-id>`.
- Reasoning text is multi-sentence and the confidence is not always
  `0.78` (the offline fixture's constant).
- The large refund returns `approval_required` on the first attempt.
- Granting the approval makes the same retry execute and lands the
  order in `REFUNDED`.
- Denying a different approval keeps the retry in `approval_required`
  and leaves the order in `PAID`.
- The rebuild check at the end matches materialized state for every
  order.

## What ran, what waited, and why

| Ticket | Amount      | KIFF action chosen | How the boundary handled it                   | Final state          |
| ------ | ----------- | ------------------ | --------------------------------------------- | -------------------- |
| 1      | 4 200 cents | `AUTO_REFUND`      | under the auto-refund ceiling (10 000 cents)  | `REFUNDED`           |
| 2      | 99 900 cents | `REFUND_ORDER`    | high-risk action; approval required           | `REFUNDED` (granted) |
| 3      | 25 000 cents | `REFUND_ORDER`    | high-risk action; approval required           | `PAID` (denied)      |

The threshold lives in routing (`RouteRefund` in `domain.go`), not in
KIFF. KIFF v0.3's `ApprovalRequirement` is binary; the contract declares
authority shape, the routing layer decides which contract a particular
agent intent lands on. This separation is the right production pattern
even if you eventually push policy into the framework.

## How the bridge actually maps

The agent registers a single tool, `refund_order`, that takes
`order_id`, `amount_cents`, and `reason`. The KIFF demo server routes
the request to one of two contracts:

- `AUTO_REFUND`  — `Risk: medium`, `ApprovalNever`, allowed in `PAID`.
- `REFUND_ORDER` — `Risk: high`,   `ApprovalRequired`, allowed in `PAID`.

Plus two ancillary contracts the demo seeds and the multi-tool fear
covers in spirit:

- `MARK_PAID`  — used at seed time to put orders in `PAID`.
- `WAIVE_FEE`  — same approval shape as `REFUND_ORDER`, exposed via
  `POST /demo/agent/waive` for buyers who want to see a second sensitive
  action plug into the same governance shape.

The demo HTTP server is essentially a translator. The agent never sees
the runtime; it only sees stable outcome strings (`executed`,
`approval_required`, `permission_denied`, `state_not_allowed`,
`missing_parameter`, `blocked`). This is the same contract used by
[`examples/llm-bridge`](../llm-bridge/), and what every model-facing
tool call should expose.

## Endpoints

Demo-only routes added by this example's server:

```
GET  /demo/orders                     list seeded orders + state
POST /demo/agent/refund               agent's "refund_order" tool
POST /demo/agent/waive                agent's "waive_fee" tool
GET  /demo/rebuild?entity=<id>        rebuild + materialized comparison
```

Standard kiff `httpapi` routes are also exposed unchanged:

```
POST /events/raw
GET  /entities/{id}/timeline
GET  /entities/{id}/allowed-actions
POST /entities/{id}/actions/{action}/validate
POST /entities/{id}/actions/{action}/execute
POST /entities/{id}/actions/{action}/approvals
POST /approvals/{id}/grant
POST /approvals/{id}/deny
GET  /admin
```

## Why this is the headline demo

- **The buyer reads it in five lines.** The agent issues refunds. The risky
  one waits for a human. The repeat is declined. The trail replays. No domain
  glossary required.
- **Enablement is the sale.** The agent does the real revenue/ops work — the
  boundary is what lets you put it on that path. The approval and the decline
  are what make shipping it safe, not the point of the demo.
- **It is reproducible without keys.** A CTO can run it on a plane.
- **It shows the same agent, same prompt, same model.** The diff is the
  boundary, not the agent's intelligence.
- **The proof lands at the end.** Three timelines, three rebuilds, zero drift.
  That is the answer to "why did this happen three weeks ago?".
